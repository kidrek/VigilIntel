## Campagne de Device Code Phishing exploitant la page légitime microsoft[.]com/devicelogin

### Résumé

Securelist (Kaspersky) rapporte une campagne de phishing où les attaquants incitent les victimes à saisir un code d'autorisation directement sur la page officielle microsoft[.]com/devicelogin (Microsoft Identity Platform). Le mécanisme abusé est le Device Authorization Grant (OAuth 2.0), conçu pour les appareils sans clavier (TV, IoT, imprimantes). L'attaquant initie une requête /oauth2/v2.0/devicecode avec un client_id malveillant, récupère un user_code, et pousse la victime à l'entrer sur la page Microsoft de confiance. Une fois le code validé par la victime, l'attaquant obtient un jeton d'accès au tenant Entra ID et donc aux ressources Microsoft 365 de la victime (mail, fichiers, Teams, OneDrive). L'attaque contourne les conseils classiques anti-phishing puisque l'URL saisie par la victime est légitime.

---

### Analyse opérationnelle

Pour les équipes SOC/IT, l'impact majeur est que les solutions traditionnelles de filtrage URL/web ne détectent rien car la victime interagit avec un domaine Microsoft officiel. La détection doit reposer sur : (1) la journalisation Entra ID avec corrélation des requêtes /devicecode, du client_id et des IP/source inhabituelles, (2) la surveillance des app consentements et des grant_type=urn:ietf:params:oauth:grant-type:device_code, (3) la détection de sessions où l'IP ayant saisi le code diffère de celle ayant initié le devicecode (géo-anomalie). Les actions techniques prioritaires : restreindre ou désactiver le Device Code Flow via Conditional Access, imposer la MFA résistante au phishing (FIDO2), auditer les client_id enregistrés dans le tenant, activer la révocation automatique des jetons et surveiller les règles inbox/redirecteurs Outlook créés post-compromission.

---

### Implications stratégiques

Cette technique illustre l'évolution du phishing vers l'abus de protocoles d'identité légitimes (OAuth Device Code, Consent Phishing, Illicit Consent Grant), rendant obsolète la seule vérification d'URL. La conséquence décisionnelle est la nécessité de repenser la stratégie IAM Microsoft : interdiction du Device Code Flow si non requis, durcissement des Conditional Access, gouvernance des applications OAuth tierces, et formation des utilisateurs au-delà du simple « vérifier l'URL ». Sectoriellement, toute organisation dépendante de Microsoft 365 (pratiquement universelle en entreprise) est exposée, avec un risque élevé de fuite de données, d'BEC (Business Email Compromise) et d'exfiltration de boîtes mail. Le risque réglementaire RGPD est significatif en cas d'accès à des données personnelles via mail ou SharePoint.

---

### Recommandations

* Restreindre le Device Code Flow aux appareils managés via Conditional Access, ou le désactiver globalement.
* Imposer la MFA résistante au phishing (FIDO2 / Windows Hello) pour toute connexion Entra ID.
* Auditer et inventorier tous les client_id et applications OAuth enregistrés dans le tenant.
* Former les utilisateurs à ne jamais saisir de code microsoft[.]com/devicelogin sans contexte attendu (smart TV, IoT).
* Centraliser les logs Entra dans le SIEM et créer des détections Sigma sur le grant_type device_code.
* Révoquer régulièrement les refresh tokens et surveiller la création de règles Outlook post-connexion.

---

### Playbook de réponse à incident

#### Phase 1 — Préparation

* Cartographier toutes les applications OAuth enregistrées dans Microsoft Entra ID et lister les redirect_uri, scopes et client_id autorisés.
* Activer et imposer l'authentification multifacteur (MFA) résistante au phishing (FIDO2, Windows Hello, Certificate-based) pour tous les utilisateurs.
* Configurer les Conditional Access policies pour restreindre le Device Code Flow aux appareils managés (compliant/hybrid joined) ou désactiver le flux si non utilisé.
* Former les utilisateurs et le helpdesk à reconnaître les demandes de code d'appareil inattendues et à signaler les connexions initiées depuis microsoft[.]com/devicelogin sans contexte.
* Déployer une journalisation centralisée des logs Entra ID (Sign-in logs, Audit logs) vers le SIEM avec rétention ≥ 90 jours.

#### Phase 2 — Détection et analyse

* Détecter les requêtes POST vers /oauth2/v2.0/devicecode provenant d'IP inhabituelles (Tor, VPN résidentiels, géolocalisation anormale).
* Alerter sur les Device Code Flows initiés par des applications non approuvées ou par des client_id inconnus (corrélation avec l'inventaire Entra).
* Détecter les sessions Microsoft Entra où le user_code est saisi depuis une IP distincte de celle ayant initié la requête /devicecode (géo-anomalie).
* Surveiller les pics de polling vers /oauth2/v2.0/token et les succès de connexion immédiatement suivis d'activités inhabituelles (exfiltration mailbox, création de règles inbox).
* Miser en place de règles Sigma/UEBA sur les nouveaux app consentements et les sessions impliquant le devicecode grant_type.

#### Phase 3 — Confinement, éradication et récupération

* Révoquer immédiatement les jetons (refresh tokens) et sessions de l'utilisateur ciblé via Microsoft Entra ID (Revoke-AzureADUserAllRefreshToken / Graph API).
* Désactiver temporairement le Device Code Flow via Conditional Access pour les applications légitimes suspectées compromises.
* Bloquer le client_id malveillant dans Entra ID et retirer tout consentement applicatif associé.
* Isoler le poste de l'utilisateur ayant saisi le code (EDR, désactivation compte AD) pour analyse forensique.
* Rechercher et supprimer les règles Outlook malveillantes, redirecteurs et applications OAuth consenties par la victime.

#### Phase 4 — Activités post-incident

* Analyser la chaîne de compromission complète : vecteur initial (phishing, social engineering), durée d'exposition, données accessibles (mail, fichiers, Teams).
* Vérifier l'inventaire des applications Entra et retirer tout consentement non légitime ou orphelin.
* Communiquer aux parties prenantes et, si requis, notifier les autorités (CNIL, ANSSI) et les clientsimpactés sous 72h (RGPD Art. 33).
* Produire un rapport de lessons learned et mettre à jour les playbooks, les politiques Conditional Access et les modules de sensibilisation.
* Vérifier l'absence de pivot latéral : revue des logs M365, activité SharePoint/OneDrive, accès Teams, etablissement d'un timeline forensique.

#### Phase 5 — Threat Hunting (proactif)

* Chasser proactivement les connexions Entra utilisant le grant_type 'urn:ietf:params:oauth:grant-type:device_code' sur les 90 derniers jours.
* Rechercher les app consentments inhabituels et les client_id jamais vus dans le tenant.
* Corréler les logs de polling /token avec les logs réseau (proxy, DNS) pour identifier les infrastructures d'attaque.
* Chasser les IOC associés : domaines de phishing, infrastructures ayant initié les requêtes /devicecode, User-Agents atypiques.
* Tester régulièrement des attaques simulées de Device Code Phishing (purple team) et mesurer la détection/réponse SOC.

## Le ransomware devient la première cause de fuites de données dans le retail (Verizon DBIR 2026)

### Résumé

Selon l'étude Verizon 2026 relayée par Indigo Privacy, le ransomware représente désormais 32 % des déclarations de fuites de données chez les détaillants et concentre 61 % des pertes financières associées. Chaque point de vente stocke des données de paiement et personnelles, qui constituent la cible privilégiée des attaquants.

---

### Analyse opérationnelle

Les équipes SOC retail doivent prioriser la surveillance des chemins d'attaque menant aux bases clients et aux systèmes POS, ainsi que des signes précurseurs ransomware (désactivation de sauvegardes, création de comptes, déplacement latéral). La segmentation réseau POS/back-office, l'EDR sur les terminaux et l'authentification forte sur les accès d'administration distants deviennent critiques. Les sauvegardes doivent être testées régulièrement et isolées pour résister à la double extorsion.

---

### Implications stratégiques

Le secteur retail, historiquement exposé par son parc hétérogène en magasin et ses sous-traitants, devient la cible financière la plus rentable pour les opérateurs de ransomware. La concentration des pertes (61 %) sur 32 % des incidents traduit un risque systémique : un seul incident peut compromettre des millions de clients et entraîner des coûts réglementaires (RGPD), contractuels (PCI DSS) et réputationnels majeurs. Les directions retail doivent intégrer ce risque dans leur gouvernance cyber et leur stratégie d'assurance.

---

### Recommandations

* Auditer la segmentation réseau POS/back-office et corriger les chemins de mouvement latéral.
* Tester la restauration des sauvegardes dans un environnement isolé et chronométrer le RTO/RPO.
* Déployer une authentification forte résistante au phishing pour tous les accès d'administration retail.
* Cartographier les données personnelles détenues par chaque enseigne et appliquer la minimisation.
* Renforcer les clauses de sécurité avec les intégrateurs et fournisseurs POS.

---

### Playbook de réponse à incident

#### Phase 1 — Préparation

* Maintenir un inventaire exhaustif des systèmes de paiement (POS, e-commerce, passerelles) et des données personnelles stockées (PCI DSS scope).
* Segmenter strictement le réseau retail (POS/logistique/bureau) et sauvegardes isolées (air-gapped ou immuables).
* Préparer des playbooks de notification clients et de conformité (RGPD, CCPA) en cas de compromission de données de paiement.
* Sensibiliser les équipes magasin à l'hameçonnage et à la sécurité physique des terminaux.
* Déployer EDR sur les hôtes gérant les flux de paiement et le back-office.

#### Phase 2 — Détection et analyse

* Surveiller les schémas d'exfiltration depuis les bases clients/CRM et les volumes inhabituels depuis les POS.
* Détecter les indicateurs de compromission ransomware (shadow copies supprimées, services de sauvegarde arrêtés, notes de rançon).
* Corréler les alertes sur les comptes à privilèges du siège avec les accès aux datastores retail.
* Intégrer les IOC de la Verizon DBIR 2026 dans les plateformes de threat intel et SIEM.

#### Phase 3 — Confinement, éradication et récupération

* Isoler immédiatement les magasins/environments POS impactés du reste du réseau.
* Suspendre les flux vers les processeurs de paiement le temps de l'investigation.
* Révoker les identifiants administrateur et clés d'API utilisés par les systèmes touchés.
* Activer le plan de communication (clients, banques acquéreurs, autorités de protection des données).
* Préserver les preuves (images forensiques) avant toute remédiation.

#### Phase 4 — Activités post-incident

* Évaluer la proportion de données clients compromises et publier les notifications réglementaires.
* Mener une revue post-incident ciblée sur les chemins d'attaque initiaux (phishing, RDP exposé,供应链).
* Renforcer le MFA, le patching et la segmentation POS.
* Mettre à jour les contrats cyber-assurance au regard de l'exposition ransomware retail.
* Participer aux communautés sectorielles (RH-ISAC) pour partager les IOC et TTP.

#### Phase 5 — Threat Hunting (proactif)

* Rechercher des signes de dwell time long (comptes dormants réactivés, services non autorisés) dans les environnements retail.
* Chasser les outils de double extorsion (Cobalt Strike, SystemBC, BazarLoader) sur les sous-réseaux magasin.
* Auditer les accès tiers (fournisseurs POS, intégrateurs) et la persistance (tâches planifiées, services).
* Tester la résilience via simulations de ransomwares ciblant les sauvegardes retail.
