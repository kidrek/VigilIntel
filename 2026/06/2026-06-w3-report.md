## Strategic Recommendations

1. **Enforce FIDO2 Hardware MFA for Developer Accounts**
   Traditional SMS or TOTP MFA is trivially bypassed by advanced infostealers like Lumma v17 and automated smishing infrastructures like Outsider Enterprise. Mandate the use of FIDO2/WebAuthn-compliant hardware security keys (e.g., YubiKey 5 Series) for all accounts with access to GitHub, AWS, and enterprise identity providers (Okta, Azure AD). Ensure session lifetime restrictions are strictly applied to mitigate the risk of stolen session cookies.

2. **Deploy Immutable Air-Gapped Cloud Backups**
   Recent wiper campaigns have proven that attackers who compromise cloud or hypervisor consoles systematically delete online and replication backups to enforce total network erasure. Implement write-once-read-many (WORM) immutable cloud storage policies combined with an air-gapped, physical backup architecture. Require multi-party approval (dual control) for any backup modification or deletion request.

## Operational Recommendations

1. **Audit GitHub Actions Workflow Triggers immediately**
   The Shai-Hulud supply chain worm actively exploits misconfigured pull_request_target triggers. Scan all organization repositories using tools like Semgrep or static analysis configurations to identify workflows using pull_request_target combined with an explicit checkout of untrusted code. Restrict permissions to read-only for tokens generated by pull requests from forks.

2. **Block PostgreSQL Sidecar Ports on Splunk Enterprise**
   To mitigate the immediate risk of CVE-2026-20253 (CVSS 9.8 RCE) in Splunk Enterprise, isolate and restrict all network access to the internal PostgreSQL sidecar port (typically 5432 or adjacent custom ports). Apply the vendor-provided hotfix immediately. Limit network communication to Splunk management consoles to authorized administrator VLANs and VPN gateways only.

## Recommandations Stratégiques

1. **Imposer l'authentification MFA matérielle FIDO2 pour les développeurs**
   Le MFA traditionnel par SMS ou TOTP est facilement contourné par des infostealers comme Lumma v17 et des infrastructures de smishing automatisées comme Outsider Enterprise. Imposez l'utilisation de clés de sécurité matérielles conformes à FIDO2/WebAuthn (ex: YubiKey série 5) pour tous les comptes ayant accès à GitHub, AWS et aux fournisseurs d'identité d'entreprise.

2. **Déployer des sauvegardes cloud immuables et isolées**
   Les récentes campagnes de wipers ont prouvé que les attaquants qui compromettent les hyperviseurs suppriment systématiquement les sauvegardes en ligne. Implémentez des politiques de stockage cloud immuable de type WORM (Write-Once-Read-Many) combinées à une architecture de sauvegarde isolée physiquement et logiquement (air-gapped).

## Recommandations Opérationnelles

1. **Auditer immédiatement les déclencheurs de workflows GitHub Actions**
   Le ver Shai-Hulud exploite les déclencheurs pull_request_target mal configurés. Scannez tous les dépôts de l'organisation à l'aide d'outils comme Semgrep pour identifier les workflows utilisant pull_request_target combiné avec un checkout de code non approuvé. Limitez les jetons générés par les pull requests externes en lecture seule.

2. **Bloquer les ports sidecar PostgreSQL de Splunk Enterprise**
   Pour atténuer le risque immédiat de la CVE-2026-20253 (RCE CVSS 9.8), isolez et restreignez l'accès réseau au port sidecar PostgreSQL interne de Splunk Enterprise (généralement 5432). Appliquez immédiatement le correctif fourni par l'éditeur et limitez l'accès d'administration aux seuls VLANs autorisés.
