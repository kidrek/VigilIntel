## Trends & Weak Signals

**Emerging techniques:**
- Abuse of trusted cloud-based artifact signing platforms (e.g. Azure Trusted Signing) for malware distribution
- OAuth 2.0 Device Code phishing bypasses of MFA via legitimate endpoint login loops
- Exploitation of custom named extensions (VMAccess naming chaos) to establish persistent administrative access on Azure VMs
- BYOVD (Bring Your Own Vulnerable Driver) deployments utilizing Microsoft-signed vulnerable drivers to terminate EDR processes

**Declining techniques:**
- Traditional SMS or TOTP based multi-factor authentication due to rapid, automated session hijacking
- Static signature-based detection for newly compiled open-source dependencies (e.g., Go/Rust binaries)
- Perimeter-only firewalls lacking microsegmentation or behavior monitoring (drift detection) inside cloud execution runtimes

## Strategic Recommendations

1. **Enforce FIDO2 Hardware MFA for Developer Accounts**
   Traditional SMS/TOTP MFA is routinely bypassed by EvilTokens and Device Code phishing (used actively by Lazarus, Scattered Spider, and Tycoon2FA this period). Deploy FIDO2/WebAuthn hardware keys (such as YubiKey 5 or Google Titan) for all developer and administrative accounts with access to cloud infrastructure and CI/CD pipelines. This mitigates the risk of session hijacking and credential theft.

2. **Establish Secure Mirroring for Open Source Dependencies**
   Software supply chain compromise remains a critical vector, as evidenced by TeamPCP's campaign targeting over 540 npm packages using the Shai-Hulud worm. Implement an internal package mirroring repository (using Nexus or Artifactory) with automated vulnerability and typosquatting scanning (using tools like OSV-Scanner). Disable direct download of newly published public dependencies to prevent the execution of malicious lifecycle scripts.

## Operational Recommendations

1. **Disable Azure VM Guest Password Reset Features**
   Attackers exploit VMAccess naming chaos (manipulating the ARM Microsoft.Compute/virtualMachines/extensions/write API) to establish silent persistence. Audit Azure environment configurations and enforce SSH key-only authentication on all Linux workloads. Explicitly monitor discrepancies between the extension 'type' and 'name' properties within Azure Activity Logs using KQL queries.

2. **Block Execution of the Windows MSHTA Utility**
   The native LOLbin mshta.exe is heavily abused by loaders like CountLoader and Emmenhtal to inject LummaStealer in memory. Deploy WDAC (Windows Defender Application Control) or AppLocker policies to globally restrict the execution of mshta.exe across user workstations. Implement ASR (Attack Surface Reduction) rules to block child processes generated by script-hosting utilities.

## Recommandations Stratégiques

1. **Imposer le MFA matériel FIDO2 pour les développeurs**
   Le MFA traditionnel par SMS/TOTP est contourné par les techniques d'EvilTokens et d'hameçonnage Device Code (utilisées activement par Lazarus, Scattered Spider et Tycoon2FA). Déployez des clés matérielles FIDO2/WebAuthn (YubiKey 5, Google Titan) pour tous les comptes de développement et d'administration accédant aux infrastructures cloud et aux pipelines CI/CD afin de bloquer le vol de sessions.

2. **Établir un miroir sécurisé pour les dépendances open source**
   La compromission de la chaîne d'approvisionnement reste un vecteur critique, comme l'illustre la campagne de TeamPCP ciblant plus de 540 paquets npm avec le ver Shai-Hulud. Implémentez un dépôt miroir interne (via Nexus ou Artifactory) avec détection automatisée des vulnérabilités et du typosquattage (via OSV-Scanner) et désactivez le téléchargement direct public des dépendances récentes.

## Recommandations Opérationnelles

1. **Désactiver la réinitialisation de mot de passe Azure**
   Les attaquants exploitent le chaos de nommage de VMAccess (en manipulant l'API ARM Microsoft.Compute/virtualMachines/extensions/write) pour établir une persistance furtive. Auditez les configurations Azure et imposez l'authentification exclusive par clés SSH sur Linux. Surveillez les écarts entre les propriétés 'type' et 'name' d'extension dans les Azure Activity Logs via KQL.

2. **Bloquer l'exécution de l'utilitaire Windows MSHTA**
   L'outil natif mshta.exe (LOLbin) est massivement détourné par des chargeurs comme CountLoader et Emmenhtal pour injecter LummaStealer en mémoire. Déployez des règles WDAC ou AppLocker pour interdire l'exécution de mshta.exe sur les postes de travail des utilisateurs et configurez des règles ASR pour bloquer la création de processus fils par des interpréteurs de scripts.
